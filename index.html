書心減重診所｜預約系統完整規格

給 Antigravity 的 Google Sheets + Apps Script 預約表設計稿

本版已依你的補充需求重新整理成可直接進入設計與開發的規格，重點納入 醫師、諮詢師、營養師三種會接觸客人的角色、多診次設定、 月份／週次／單日三種行事曆視圖，並補上診所實務中常被漏掉的 資源衝突、報到狀態、候補、回診、黑名單、提醒、權限、統計、Google Sheets 結構。

核心目標
  • 讓櫃台可快速建立、改期、取消、查詢與追蹤預約。
  • 讓診所可設定每位人員、每個診次、每種服務的可約規則。
  • 讓月曆、週曆、日曆畫面都能依時間順序清楚顯示。
  • 讓 Google Sheets 同時扮演資料庫、設定中心與簡易報表來源。
本次修正重點
  • 修正原方案偏一般預約系統，未完整反映減重診所流程。
  • 補上「診1、診2、診3」與每診次分鐘長度的後台設計。
  • 補上不同角色是否需同時出現在同一筆預約的規則。
  • 補上 Antigravity 可直接照做的欄位與畫面規格。

一、業務場景重新定義

角色

接觸客人的服務人員

至少包含 醫師、諮詢師、營養師。未來若要增加護理師、個管師、抽血室、體雕師，也要能用同一套設定擴充。

資源

多診次、多房、多時段

後台可建立 診1、診2、診3 等診次／診間資源，每個診次可各自設定可約時段與每位客人的服務分鐘數。

流程

不是只有看診，還包含前後段服務

一位客人的一次來診可能包含 諮詢 → 醫師 → 營養師，也可能只約單一服務，因此系統必須支援單段與多段預約。

關鍵設計原則：預約系統不能只記「某天某時間有某客戶」，而是要同時管理 客戶、服務類型、人員、診次資源、時長、狀態、提醒、來源、回診關聯。

二、建議的完整預約模型

A. 預約主檔 Appointment

  • 一筆主預約代表客人某日來院的一次完整行程。
  • 主預約底下可拆成多個服務段，例如「初診諮詢 30 分 + 醫師 15 分 + 營養 30 分」。
  • 適合做取消、改期、回診追蹤、報表統計。

B. 預約明細 Appointment Segment

  • 每個 Segment 對應一段實際占用時間與資源。
  • 每段都可指定服務角色、人員、診次、開始時間、結束時間、時長。
  • 適合處理「同一客戶一次來院會輪流見不同人」的需求。
為什麼一定要拆主檔 + 明細？ 因為減重診所常見同日多段流程。如果只做一張單表，之後會很難處理： 一位客戶同一天看醫師後再接營養師、不同段使用不同診次、不同段時長不同、其中一段取消但另一段保留。

三、Google Sheets 建議資料表

1. Clients 客戶主檔

儲存客戶基本資料、聯絡方式、來源、特殊提醒與風險註記。

2. Staff 員工主檔

儲存醫師、諮詢師、營養師等人員，並記錄角色、可排班狀態、顏色、是否可被預約。

3. Sessions 診次設定

定義診1、診2、診3等資源，可設定日別開放時間、分鐘長度、同時容納量。

4. Service Types 服務類型

例如初診諮詢、醫師複診、營養衛教、體重追蹤、療程解說、回診抽血等。

5. Appointments 預約主檔

記錄主預約編號、客戶、日期、來源、總狀態、備註、提醒、回診關聯。

6. Appointment Segments 預約明細

記錄每一段服務的時間、人員、診次、角色、時長與狀態。

7. Closures 公休/停診

設定診所公休、醫師停診、教育訓練、颱風假、特殊封鎖時段。

8. Waitlist 候補名單

當熱門時段額滿時，保留候補客戶、偏好時段、可通知方式。

9. Audit Log 稽核日誌

記錄誰在什麼時間新增、修改、取消、恢復或調整預約。

四、每張 Sheet 的關鍵欄位

Sheet 必要欄位 說明
Clients client_id、姓名、手機、生日、性別、來源、首次來院日、備註、黑名單註記、重要提醒 重要提醒例如「需電話確認」「不接受晚間」「對特定營養師有偏好」。
Staff staff_id、姓名、角色、可預約、顏色、排序、在職狀態、可服務項目 角色至少支援 doctor / consultant / dietitian，也可一人多角色。
Sessions session_id、名稱、類型、起始時間、結束時間、預設分鐘、容量、適用角色、適用星期、啟用狀態 例如診1：09:00-12:00，15 分鐘 / 人；診2：14:00-17:30，30 分鐘 / 人。
Service Types service_type_id、名稱、預設角色、預設時長、是否初診、是否需診次、是否需醫師、是否需營養師、顏色 讓不同服務自動帶出時長與應配置人員。
Appointments appointment_id、client_id、預約日期、預約來源、主狀態、主備註、建立者、建立時間、回診批次、提醒狀態 主狀態例如已預約、已報到、已完成、改期、取消、爽約。
Appointment Segments segment_id、appointment_id、sequence、service_type_id、role、staff_id、session_id、開始、結束、分鐘數、segment_status sequence 用來排順序，例如 1 諮詢、2 醫師、3 營養師。
Closures closure_id、日期、開始、結束、對象類型、對象 id、原因、是否阻擋預約 可針對整院、特定人員、特定診次封鎖。
Waitlist waitlist_id、client_id、期望日期、期望時段、期望角色、優先級、通知方式、狀態 有人取消時可自動篩選候補客戶。

五、後台設定規格：診1、診2、診3 與分鐘長度

診次基本設定

  • 名稱：診1 / 診2 / 診3 / 營養室A / 諮詢室B。
  • 資源類型：診間、諮詢室、營養室、共用時段。
  • 啟用狀態：啟用 / 停用。
  • 顯示排序：決定日曆欄位先後。

時間切片設定

  • 預設分鐘：15、20、30、45、60 分鐘。
  • 可依診次不同，例如診1 每 15 分，診2 每 30 分。
  • 可依服務類型覆蓋預設值，例如初診固定 60 分鐘。
  • 可設定緩衝時間，例如每位客人後保留 5 分鐘。

容量與衝突規則

  • 單診次同一時間可約幾位客人。
  • 是否允許超掛。
  • 是否允許同一人員跨診次重疊。
  • 是否限制同一客戶不可同時重複預約。
建議公式： 可約時段 = 診次開放區間 ÷ 單位分鐘，若有緩衝時間，實際切片 = 服務分鐘 + 緩衝分鐘。 Apps Script 應在產生可預約清單時一併扣除已占用時段與停診時段。

六、預約建立規則

必填欄位

  • 客戶姓名或客戶編號
  • 手機
  • 預約日期
  • 服務類型
  • 至少一段預約明細
  • 主責人員或由系統自動指派

自動檢查

  • 同客戶同時間不可重複預約。
  • 同 staff_id 同時間不可撞單。
  • 同 session_id 超過容量不可預約。
  • 停診、公休、封鎖時段不可預約。
  • 黑名單客戶需跳警示，是否允許預約由管理員設定。

常見預約模板

情境 建議 Segment 總時長示例 說明
初診減重評估 諮詢師 30 → 醫師 15 → 營養師 30 75 分鐘 最常見的一次到院多段流程，建議做成一鍵套用模板。
醫師複診 醫師 15 15 分鐘 適合高頻大量排程。
營養追蹤 營養師 30 30 分鐘 可獨立預約，不一定需要醫師。
療程說明／方案調整 諮詢師 30 → 醫師 15 45 分鐘 看實務可再加護理站或檢測段。

七、UI 畫面規格

月份日曆

Month View

必須以星期日為第一天，用月曆格狀呈現。每格顯示該日總預約數、醫師數、待確認數、取消數，點格子可展開當日清單。

  • 上方可切換月份。
  • 格內依時間順序列出摘要。
  • 不同角色 / 狀態以顏色區分。
週次格式

Week View

以七天橫向、時間縱向呈現，適合看整週排程。支援欄位切換成「依人員」、「依診次」或「依角色」分欄。

  • 可快速拖曳改時間。
  • 可看衝堂與空檔。
  • 適合櫃台調度與主管檢視量能。
單日格式

Day View

以單日時間軸呈現，可依 5 / 10 / 15 / 30 分鐘刻度顯示。是最重要的作業畫面，適合現場櫃台即時操作。

  • 依時間排序。
  • 可顯示客戶、服務類型、診次、人員、狀態。
  • 可一鍵報到、改期、取消、完成。
畫面建議： 若用 Apps Script HTML Service，可在前端採用原生 HTML/CSS/JS，或搭配 FullCalendar 類型的日曆元件概念實作。 若 Antigravity 想降低開發成本，可先做「月/週/日切換 + 右側詳情抽屜」的單頁介面。

八、操作流程規格

1. 櫃台搜尋客戶：可用姓名、手機末四碼、客編搜尋。
2. 若是新客戶，先建立 Clients 主檔。
3. 選擇服務模板或手動新增多個 Segment。
4. 系統依服務類型與角色，顯示可用人員與可用診次。
5. 系統檢查衝突、停診、容量、黑名單、重複預約。
6. 建立主預約與明細，寫入 Sheets，並同步產生日曆畫面資料。
7. 可選擇立即發送 LINE / 簡訊 / Email / 電話提醒註記。
8. 到診後切換狀態：已報到 → 進行中 → 已完成，或改為取消 / 改期 / 爽約。

九、你原本可能沒想到，但診所很需要的功能

1. 候補名單

熱門醫師或晚間時段一定會需要，取消時可立即補位。

2. 爽約與取消原因

用來分析是否某時段、某來源、某服務特別容易流失。

3. 回診週期建議

例如醫師看完後建議 2 週回診，可自動帶出建議日期。

4. 黑名單 / 高風險提醒

例如經常爽約、情緒衝突、付款異常，需要櫃台先看到警示。

5. 到診統計

不是只有預約數，還要看報到率、完成率、爽約率、改期率。

6. 服務量能管理

看每位醫師／營養師／諮詢師每日可服務幾位、目前已排幾位。

十、Apps Script 後端功能清單

API / Function 建議

  • getCalendarView(viewType, startDate, endDate, filters)
  • searchClients(keyword)
  • createClient(payload)
  • createAppointment(payload)
  • updateAppointment(appointmentId, payload)
  • cancelAppointment(appointmentId, reason)
  • checkAvailability(payload)
  • getSettings()
  • saveSessionSettings(payload)
  • writeAuditLog(action, payload)

必要機制

  • 使用 LockService 避免多人同時寫入撞單。
  • 使用 CacheService 快取員工、服務、診次設定。
  • 所有新增／修改／取消都寫入 Audit Log。
  • 寫入前先跑 availability 檢查。
  • 建立每日提醒、次日提醒、未到提醒批次。

十二、權限分級建議

管理者 Admin

可維護所有設定、人員、診次、停診、公休、報表與權限。

櫃台 Front Desk

可新增、修改、取消、報到、搜尋與查看全院預約，但不能改核心設定。

專業人員 Staff View

醫師、諮詢師、營養師可查看自己或授權範圍內的行程。

十三、Antigravity 開發交付建議

  1. 先完成 Google Sheets 結構與欄位命名，不要先寫畫面。
  2. 再完成 Apps Script 的資料存取層與 availability 檢查。
  3. 先上 Day View，因為櫃台最常用；再做 Week View；最後補 Month View。
  4. 先支援三個主要角色：醫師 / 諮詢師 / 營養師，其他角色保留擴充欄位。
  5. 先做 4 個核心狀態：已預約、已報到、已完成、取消；第二階段補改期與爽約。
  6. 先做手動排程，第二階段再做自動推薦空檔與候補補位。
最務實的 MVP：客戶主檔 + 員工主檔 + 診次設定 + 預約主檔 + 預約明細 + 月/週/日三視圖 + 改期取消 + 衝突檢查。

十四、最後結論

這套系統的關鍵，不是做一張漂亮日曆而已，而是把診所實際運作邏輯放進資料模型： 誰服務、在哪個診次、服務多久、是否能重疊、是否停診、是否需要多段流程、如何追蹤到診與回診。

若 Antigravity 依本規格開發，建議直接以 Google Sheets 作為設定與資料層、 Apps Script 作為邏輯與 API 層、HTML Service 作為 UI 層，即可快速完成第一版。

最重要提醒：請不要只做單一預約表。一定要拆成「主預約 + 預約明細 + 診次設定 + 人員設定」， 否則之後遇到醫師／諮詢師／營養師串接流程時，資料會很快失控。

本頁為書心減重診所預約系統規格整理版，可直接提供 Antigravity 進入 Google Sheets 與 Apps Script 設計。
